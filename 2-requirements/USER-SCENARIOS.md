# FounderOS 用户场景

*从 Founder（Jeff）的视角描述日常使用场景。每个场景回答"用户做什么 → 系统做什么 → 用户看到什么"。*

---

## 场景 1：早晨看全局

**触发**：Jeff 早上打开系统（或对 Telegram Bot 说"状态"）

**用户意图**：5 分钟内知道公司现在什么情况，有没有需要立刻处理的事

**系统行为**：
```
1. 脊髓引擎生成 L0 看板（全部活跃根节点 + risk marks）
2. L_Alert 列出所有 blocked / at_risk / stale 节点
3. 如果有焦点节点，附带 L1 摘要
```

**用户看到的**：
```
📊 公司看板 (2026-03-20)

📁 FounderOS [active] ✅ on_track
  ├─ FPMS [active] ✅ v1 完成，已接入
  ├─ CTO Agent [active] ⚠️ at_risk — deadline 3天后，PRD待确认
  └─ 支付系统 [inbox]

⚠️ 需要关注:
  • CTO Agent: deadline 3/23，当前状态"PRD V2完成，待搭建"，无子任务分解
  • 供应商对接: blocked by 合规审批（已等待 5 天）

🎯 当前焦点: CTO Agent
```

**用户下一步**：决定要不要 shift_focus 到某个告警项，或继续当前焦点

---

## 场景 2：深入一个任务

**触发**：Jeff 说"CTO Agent 现在什么情况"或 shift_focus 到某个节点

**用户意图**：深入了解这个任务的完整状态，做出决策

**系统行为**：
```
1. 脊髓引擎为目标节点组装 Context Bundle（L2 焦点）
2. 装载：自身骨架 + 叙事 + 子任务状态 + 依赖 + 父节点目标
3. 进入校验：确认任务与父目标对齐
```

**用户看到的**：
```
🎯 CTO Agent [active]

为什么做: FounderOS 需要第一个 Office 实例来验证 Memory Architecture
当前状态: PRD V2 完成，待搭建
下一步: 拆解为子任务，开始 scaffold
截止日期: 2026-03-23 ⚠️ 3天后

子任务: (无 — 需要拆解)

最近叙事:
  3/18: PRD V2 完成，覆盖 workspace 隔离 + FPMS 集成 + 代码审查流程
  3/15: V1 反馈整合，加入安全沙箱设计

依赖:
  ← FPMS v1 [done] ✅ 已满足
  ← Memory Architecture 设计 [done] ✅ 已满足

⚠️ 风险: 还没拆解子任务，deadline 只剩 3 天
```

**用户下一步**：告诉 Agent "拆解这个任务"或自己定义子任务

---

## 场景 3：几个月后回来问进展

**触发**：Jeff 三个月没碰某个项目，突然问"支付系统怎么样了"

**用户意图**：快速恢复对这个项目的认知，不需要重读所有历史

**系统行为**：
```
1. 支付系统节点的叙事已经过多轮压缩
2. 脊髓引擎装载: 压缩后的叙事摘要 + 当前子任务状态 + 活跃风险
3. 不需要装载三个月的详细记录，压缩摘要已包含所有关键决策和原因
```

**用户看到的**：
```
🎯 支付系统 [active]

摘要:
  经调研选定 Stripe Connect 方案（成本可控 + 多币种 + 合规内置），3月决策。
  4月完成收单模块，5月完成发卡模块。
  6月合规审批通过，正式上线。当前聚焦跨境结算模块。

当前子任务:
  ├─ 收单模块 [done] ✅
  ├─ 发卡模块 [done] ✅
  ├─ 跨境结算 [active] — 对接 SWIFT 接口中，进度 60%
  └─ 稳定币支付 [inbox] — 等跨境结算完成后启动

主要风险:
  • 跨境结算: SWIFT 沙箱环境不稳定，已报告 3 次超时

关键决策记录:
  • 3/7: 选定 Stripe Connect（淘汰方案A:不支持多币种，方案C:成本过高）
  • 5/12: 发卡模块改用预付卡方案（原信用卡方案合规成本太高）
```

**为什么能做到**：压缩机制保留了"决策+原因+当前状态+活跃风险"，丢弃了"每日对接记录、已解决的 bug、正常运转的日志"

## 场景 4：执行中发现问题

**触发**：Agent 在执行"对接 SWIFT 接口"任务时，发现 SWIFT 要求的合规认证比预期复杂

**用户意图**：（此场景不是用户主动触发，是系统主动上报）

**系统行为**：
```
1. Agent 在执行中触发中途校验（发现与父目标的假设冲突）
2. Agent 执行轻量级向上查询：拉父节点"跨境结算"的 status + summary
3. 发现这个合规要求可能影响整个跨境结算的 timeline
4. Agent 暂停执行，上报发现
```

**用户看到的**：
```
⚠️ [中途校验上报] SWIFT 接口对接

发现: SWIFT 要求 ISO 20022 合规认证，预计需要额外 3-4 周
影响: 跨境结算模块 deadline 可能延后，影响稳定币支付的启动时间
建议:
  A) 申请合规认证，接受延期
  B) 改用第三方 SWIFT 网关（成本增加但免认证）

当前状态: 已暂停，等待您的决策
```

**用户下一步**：Jeff 选 A 或 B，Agent 继续执行

## 场景 5：紧急打断

**触发**：Jeff 正在和 Agent 讨论 CTO Agent 设计，突然收到告警"支付系统生产环境报错"

**系统行为**：
```
1. Heartbeat 检测到"支付系统-生产监控"节点状态变为 blocked
2. 该节点 depended_by >= 3（多个下游依赖），触发打断条件
3. 当前焦点"CTO Agent 设计"被推入 stash
4. L_Alert 注入紧急告警
```

**用户看到的**：
```
🚨 紧急打断

支付系统-生产监控 [blocked]
  原因: Stripe webhook 回调失败，影响 3 个下游任务
  影响范围: 收单确认、对账、商户通知

⏸️ 已暂停: CTO Agent 设计 — 进展：正在讨论 workspace 隔离方案

需要您决定下一步处理方案。
```

**处理完毕后**：
```
💡 可恢复: CTO Agent 设计 — 暂停时进展：正在讨论 workspace 隔离方案
要切回吗？
```

## 场景 6：任务拆解

**触发**：Jeff 说"把 CTO Agent 拆一下"

**系统行为**：
```
1. Agent 分析 CTO Agent 的当前 context（PRD 内容、依赖关系）
2. 提出拆解方案
3. 经 Jeff 确认后，通过 create_node + attach_node 执行
```

**用户看到的**：
```
拆解方案: CTO Agent

├─ T1: Workspace 隔离 [预估 2 天]
│   搭建 CTO 独立工作目录 + memory 读写隔离
│
├─ T2: FPMS 集成 [预估 1 天]
│   接入 FPMS 读写任务状态，验证 Tool Call 全流程
│   依赖: T1
│
├─ T3: 代码审查流程 [预估 2 天]
│   PR Review + 自动化测试 + 质量门禁
│   依赖: T2
│
└─ T4: 端到端验收 [预估 1 天]
    完整场景测试：接需求 → 写代码 → 审查 → 合并
    依赖: T3

确认这个拆解吗？
```

**Jeff 确认后**：4 个子任务创建，parent 指向 CTO Agent，dependency 链自动建立

## 场景 7：决策记录

**触发**：Jeff 做了一个关键决策（如"支付系统用 Stripe Connect"）

**用户意图**：确保这个决策被记录，未来能追溯为什么这么决定

**系统行为**：
```
1. Agent 通过 append_log 记录决策
2. 记录格式: 决策内容 + 原因 + 被淘汰的方案
3. 写入节点叙事（Fact 层），不可篡改
```

**用户未来查询时看到的**：
```
关键决策 #7 (2026-03-07):
  决定: 采用 Stripe Connect
  原因: 支持多币种 + 合规内置 + 成本低于方案C
  淘汰: 方案A（不支持多币种）、方案C（年费 $50k 超预算）
  背景: 调研了 3 个方案，耗时 1 周，详见子任务"支付方案调研"
```

## 场景 8：外部工具同步（V2）

**触发**：GitHub 上有一个新的 Issue 被创建，或 Notion 文档被更新

**用户意图**：不需要手动在 FounderOS 里重复录入，系统自动感知外部变化

**系统行为（V2 认知引擎）**：
```
1. Adapter 监听 GitHub webhook / Notion API 变更
2. 变更映射为 FPMS 节点操作（create_node / update_status / append_log）
3. 脊髓引擎重算风险标记和 rollup
4. 如果变更影响当前焦点，注入 L_Alert
```

**用户看到的**：
```
📥 外部变更 (GitHub)

新 Issue: "API 响应超时 #142" → 已自动创建任务节点
  parent: 支付系统 > 收单模块
  status: inbox
  priority: 根据 GitHub labels 设置为 high

是否需要关注这个？
```

**注意**：此场景为 V2 功能，V1 阶段所有数据手动通过 Tool Call 录入。

## 场景覆盖矩阵

| 场景 | 覆盖的系统能力 | 版本 |
|------|-------------|------|
| 1 早晨看全局 | L0 看板、L_Alert、Heartbeat | V1 |
| 2 深入任务 | Context Bundle、L2 焦点、进入校验 | V1 |
| 3 几个月后回来 | 压缩、叙事摘要、认知恢复 | V1 |
| 4 执行中发现问题 | 中途校验、向上上报、暂停等待 | V1 |
| 5 紧急打断 | Stash、被动切换、打断条件 | V1 |
| 6 任务拆解 | 分形拆解、context 分区、dependency 链 | V1 |
| 7 决策记录 | 叙事写入、因果链保留 | V1 |
| 8 外部工具同步 | Adapter、webhook、自动映射 | **V2** |
