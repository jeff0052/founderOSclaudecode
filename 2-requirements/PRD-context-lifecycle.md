# PRD: Context 生命周期与任务拆解

*基于 2026-03-20 需求讨论，补充 PRD-functional-v4 的缺失环节*

---

## 核心命题

LLM 只是大脑前叶 — 针对当下 context 做推理的器官。要让它像 COO 一样工作，必须造出完整的意识结构：

| 人类意识 | 系统对应 | 职责 |
|---------|---------|------|
| 末那识（稳定人格） | Constitution（L1 记忆层） | 不变的价值观和判断框架，永远在 context 里 |
| 阿赖耶识（深层记忆） | 长期存储（SQLite + MD） | 所有历史事实、判断、叙事 |
| 大脑前叶（推理决策） | LLM | 当下 context 内的推理，不管记忆 |
| 小脑（自动化协调） | 脊髓引擎 | 自动装载、压缩、状态计算，不需要 LLM 参与 |
| 脑干（维持生命体征） | Heartbeat | 持续监控、告警，不需要意识参与 |
| 感官器官（实时输入） | Signals | 外部事件、消息、数据变化 |

**LLM 的优势不是思考，是精确存取。** 人脑压缩和判断优先级更强，但 LLM 在数据存储、调取精确度、并行检索方面远超人脑。最佳组合：人脑负责压缩、判断、优先级；LLM 负责存取、执行、一致性。

## 1. Context 的本质

Context 不是"聊天记录"，是**工作台**。

一个好 COO 被问到任何一个任务，脑子里瞬间组装的就是这个任务的工作台：

- 任务本身的状态
- 为什么要做这件事（向上）
- 做到哪了、卡在哪了（向下 + 横向）
- 上次讨论时做了什么决定（历史）

脑子里**没有**的：不相关项目的细节、三个月前的日常记录、已完成且无影响的子任务详情。

**COO 判断重要性的标准：影响决策的 = 重要，不影响任何东西的 = 不重要。**

---

## 2. 任务 Context 的组成定义

给定一个焦点任务，其 context 由以下部分组成：

| 部分 | 内容 | 何时装载 | 粒度 |
|------|------|---------|------|
| **自身骨架** | title, status, why, next_step, deadline, owner | 总是 | 全量 |
| **自身叙事** | 最近关键叙事（决策、状态变更、风险事件） | 总是 | 最近 3 天或最近 5 条 |
| **向上（为什么）** | 父节点的目标、状态、summary | 总是 | 摘要级 |
| **向下（进展）** | 子任务状态列表 | 当任务有子任务时 | 一句话状态 per 子任务 |
| **横向依赖** | 阻塞我的节点状态 + 等我的节点状态 | 当存在依赖时 | 摘要级 |
| **历史决策** | 相关的 L0/L1 决策记录 | 当需要理解"为什么这样做" | 结论 + 原因 |

**裁剪铁律**：context 超预算时，优先保留自身骨架 + why + 最近叙事（因果链），砍掉横向关系和历史细节。半残的全景不如完整的焦点。

## 3. Context 生命周期

Context 不是一次性的，是一个持续的**装载-校验-执行-校验-写回-压缩**循环。

人类做复杂任务时不是"总分总"一次，而是**反复抬头看全局**。做着做着发现新信息，立刻检查这个发现会不会影响上层目标，确认对齐后继续。系统设计必须支持这种中途校验。

```
存储层
  | 装载（精准检索，只拿当前任务需要的）
  v
Context（工作台）
  | 进入校验（焦点任务是否仍与父目标对齐？）
  v
执行（LLM 推理）──┐
  |               | 中途校验（发现重要信息时抬头看全局）
  |               | 确认对齐 → 继续执行
  |<──────────────┘
  v
产出（结论、决策、状态变更）
  | 退出上报（关键发现是否影响上层？）
  | 写回（通过 tool call 写入存储层）
  v
存储层（更新后）
  | 压缩（把详细过程抽象为摘要）
  v
存储层（压缩后）
  | 下一轮装载...
```

### 3.1 装载（Load）

由脊髓引擎执行（DCP Push），不由 LLM 检索。

- 输入：当前焦点 node_id
- 输出：组装好的 Context Bundle
- 规则：按第 2 节定义的组成装载

### 3.2 进入校验（Entry Check）

装载完成后、开始执行前，Agent 先确认焦点任务是否仍然有效。

**校验内容**：
- 父任务的目标和方向是否变化（已在 context 中，无需额外查询）
- 焦点任务的前置依赖是否已满足
- 是否有新的 L_Alert 影响当前任务

**校验结果**：
- 对齐 → 继续执行
- 不对齐 → 上报偏差，暂停等待 Founder 指令
- 前置未满足 → 标记 blocked，切换焦点

进入校验是轻量级的，基于已装载的 context 判断，不触发额外查询。

### 3.3 中途校验（Mid-flight Check）

执行过程中，Agent 可能产生影响上层判断的发现。此时需要**抬头看全局**，确认方向仍然正确。

**触发条件**（不是定时，是事件驱动）：

| 触发信号 | 说明 | 示例 |
|---------|------|------|
| 与父目标矛盾 | 发现的事实与父任务的假设冲突 | 调研发现目标市场政策已变，父任务的前提不成立 |
| 成本/范围超预期 | effort 或 scope 明显超出初始估计 | 原以为3天完成的对接，发现需要重写整个接口 |
| 发现新依赖 | 需要等待一个之前未知的前置条件 | 对接过程中发现还需要另一个团队的审批 |
| 关键假设失效 | 任务立项时的前提条件不再成立 | 选定的方案B被合作方通知下线 |

**校验动作**：

轻量级向上查询 — 只拉父节点的当前 status + summary，确认方向是否对齐。

```
触发中途校验
  ↓
get_node(parent_id) → 读取 status + summary
  ↓
对齐？
  是 → 记录校验结果，继续执行
  否 → 上报发现（append_log + 标记 needs_attention），暂停等待指令
```

**不是什么**：
- 不是定时轮询（"每 10 分钟检查一次"）
- 不是完整的 context 重装载（只拉父节点摘要）
- 不是 Agent 自己决定改方向（发现不对齐时上报，不擅自调整）

### 3.4 写回（Commit）

LLM 通过 tool call 写回产出。

**写回粒度**：只写结论，不写推理过程。

```
写回内容：
  - 状态变更: update_status(node_id, new_status, reason)
  - 关键发现: append_log(node_id, "发现 X 是根因，因为 Y")
  - 下一步: update_node(node_id, next_step="...")
  - 向上上报: append_log(parent_id, "[子任务发现] ...")  ← 新增

不写回的：
  - 中间推理过程（看了哪些节点、排除了哪些可能性）
  - 重复的状态描述
  - 对话本身的内容
  - 中途校验中确认对齐的记录（正常情况不记录，只记异常）
```

### 3.5 退出上报（Exit Report）

写回完成后，Agent 评估本次执行是否产生了**影响上层的关键发现**。

**上报条件**：执行结果改变了父任务应该关注的信息。

```
上报内容：
  - 关键发现写入父节点叙事: append_log(parent_id, "[来自 子任务title] 发现...")
  - 如果影响父任务状态: 建议状态变更（不直接改，由 Founder 或父任务的下次校验决定）

不上报的：
  - 子任务正常完成、没有意外（status rollup 已经覆盖了）
  - 过程中的细节（留在子任务叙事里）
```

**与 rollup 的区别**：rollup 冒泡的是**状态**（status_change），退出上报冒泡的是**发现**（影响判断的新信息）。两者互补。

### 3.6 压缩（Compress）

将累积的叙事抽象为摘要，让未来的 context 能延续而不爆炸。

**压缩的本质**：用更少的 token 表达同样的决策支持信息。

```
压缩前（5000 tokens）:
  - 3月1日: 调研了方案A，发现不支持多币种
  - 3月3日: 调研了方案B，技术可行但成本高
  - 3月5日: 和合作方确认方案B可以降价
  - 3月7日: 决定采用方案B
  - 3月8日: 开始对接，发现API文档过时
  - 3月10日: 拿到新文档，继续对接
  - ...

压缩后（500 tokens）:
  - 经调研选定方案B（多币种+成本可控），3月7日决策
  - 对接中发现API文档过时，已更新，当前进度：核心接口完成 3/7
  - 主要风险：合作方响应速度慢（平均48h）
```

**压缩保留什么**：决策及其原因、当前状态、未解决的风险
**压缩丢弃什么**：中间过程、已解决的问题、日常正常运转的记录

## 4. 任务拆解 = Context 分形

### 4.1 核心原则

**拆解的目的不是"管理"，是让每个子任务的 context 自给自足。**

一个任务拆解完成的判定标准：
1. 每个子任务拿起来，不需要额外翻找信息就能开工
2. 子任务的 summary + why + next_step 构成了完整的工作指令
3. 父任务只需要看子任务的状态聚合就能了解全局

### 4.2 Context 预算与估算

预算不是固定数字，按任务类型弹性调整。具体数值为系统参数，可在 `SYSTEM-CONFIG.md` 中调整：

| 任务类型 | 预算上限（默认值） | 原因 |
|---------|---------|------|
| **执行型**（写代码、处理数据、做决策） | **100k** | 需要高推理精度，噪声直接降低输出质量 |
| **分析型**（读文档、做判断、写方案） | **150k** | 推理中等，需要更多参考材料辅助判断 |
| **检索型**（找信息、对比方案、调研） | **500k** | 推理浅但需要广度，容忍大量参考材料 |

**默认预算：100k。** 只有明确判断任务属于分析型/检索型时才放宽。超过对应预算 = 必须拆解。

> **注意**：以上数值会随模型能力增强而调整。所有可调参数统一在 `SYSTEM-CONFIG.md` 管理，PRD 只定义规则，不硬编码数字。

#### 4.2.1 Context 各部分的典型 token 开销

| 部分 | 典型大小 | 说明 |
|------|---------|------|
| **固定开销** | | 每个任务都有，不可压缩 |
| Constitution（L1 宪法层） | ~1k tokens | 公司原则、审批规则，永远装载 |
| CLAUDE.md（开发规范） | ~2.5k tokens | 开发任务必须装载 |
| L0 看板（全局概览） | ~1k tokens | 所有根节点一行字 |
| L_Alert（告警） | ~0.5k tokens | blocked/at_risk/stale 列表 |
| **固定开销小计** | **~5k tokens** | |
| | | |
| **任务相关开销** | | 随任务复杂度变化 |
| 自身骨架 | ~0.5k tokens | title, status, why, next_step 等 |
| 自身叙事 | ~1-5k tokens | 最近 5 条，取决于叙事详细程度 |
| 向上（父节点摘要） | ~0.5k tokens | 一层父节点的目标 + summary |
| 向下（子任务列表） | ~0.2k × N | N = 子任务数量，每个一句话状态 |
| 横向依赖 | ~0.3k × M | M = 依赖数量，每个摘要级 |
| 历史决策 | ~0.5-2k tokens | 按需装载，结论+原因 |
| **任务相关小计** | **~3-15k tokens** | |
| | | |
| **工作载荷** | | 任务实际执行需要的参考材料 |
| PRD 相关段落 | 按实际计算 | 只装载当前任务涉及的 FR |
| 代码文件 | 按实际计算 | 需要读/写的源文件 |
| 接口签名 | 按实际计算 | 上游/下游模块的签名 |
| **工作载荷小计** | **变化极大** | 这是决定是否超预算的关键变量 |

#### 4.2.2 预算分配原则

```
预算分配（以执行型 100k 为例）:

固定开销:     ~5k   (5%)    ← 不可压缩
任务 context: ~10k  (10%)   ← 可通过压缩控制
工作载荷:     ~85k  (85%)   ← 留给实际工作的空间

分析型 150k 时:  固定 ~5k + context ~15k + 载荷 ~130k
检索型 500k 时:  固定 ~5k + context ~15k + 载荷 ~480k

如果工作载荷超过对应预算的 85% → 任务太大，必须拆解
如果任务 context 超过 15k → 叙事需要压缩
```

#### 4.2.3 估算流程

拆解前必须先估算，不是拍脑袋拆：

```
Step 1: 列出任务需要的所有参考材料（PRD段落、代码文件、接口签名）
Step 2: 估算工作载荷 token 数（1 个英文单词 ≈ 1.3 tokens，1 个中文字 ≈ 2 tokens）
Step 3: 加上固定开销（~5k）+ 任务 context（~10k）
Step 4: 总计 > 100k？
  是 → 找信息边界，拆成子任务，回到 Step 1 分别估算
  否 → 粒度合适，可以执行
```

### 4.3 拆解信号

**如果一个任务的 context 太大装不进 token 预算，说明它还没拆够。**

```
context 超预算
  -> 信号：这个任务太大了
  -> 动作：拆成子任务
  -> 结果：每个子任务的 context 变小，各自独立可工作
  -> 父任务的 context 变成子任务状态列表（rollup），也变小了
```

### 4.4 拆解规则

| 规则 | 说明 |
|------|------|
| **自给自足** | 子任务的 context（自身 + 向上一层 + 横向依赖）必须足够独立开工 |
| **不重叠** | 两个子任务的 context 不应大面积重复 |
| **可聚合** | 父任务可以通过子任务的 status + summary 理解全局，不需要深入每个子任务 |
| **停止条件** | 当一个任务的 context 在 token 预算内可完整装载时，不再继续拆解 |

### 4.5 在哪里拆：信息边界优先

拆解点不是随便选的。优先在**信息耦合最低的地方**切：

| 优先级 | 切割点 | 说明 | 示例 |
|--------|-------|------|------|
| 1 | **不同领域** | 两块内容属于不同知识域 | "数据库设计"和"前端 UI"拆成两个任务 |
| 2 | **不同文件** | 两块内容对应不同源文件 | schema.py 和 narrative.py 各一个任务 |
| 3 | **不同 PRD 段落** | 两块内容对应不同 FR | FR-1 节点模型和 FR-5 状态引擎各一个任务 |
| 4 | **不同生命阶段** | 同一个东西的设计和实现 | "设计 API"和"实现 API"拆成两步 |

**反模式：在高耦合处切。** 如果两个子任务必须互相引用大量相同的 PRD 段落和代码，说明切割点选错了，它们应该合并或换个地方切。

### 4.6 拆解后的 context 变化

```
拆解前：
  任务A（context = 15k tokens，超预算）
    - 大量细节、多个方向、复杂依赖

拆解后：
  任务A（context = 3k tokens）
    - 自身骨架 + summary
    - 子任务状态列表（rollup）：A1 进行中, A2 blocked, A3 done

  子任务A1（context = 5k tokens）
    - 自身骨架 + 叙事
    - 向上：任务A 的目标（一句话）
    - 横向：依赖 A2 的某个产出

  子任务A2（context = 4k tokens）
    - 自身骨架 + 叙事
    - 向上：任务A 的目标（一句话）
    - 横向：被 A1 依赖
```

### 4.7 谁来拆解

| 拆解者 | 场景 | 方式 |
|--------|------|------|
| Founder（人类） | L0/L1 战略级任务 | 明确告诉 Agent 怎么拆 |
| Agent（LLM） | L2/L3 执行级任务 | Agent 判断需要拆解时，提出拆解方案，通过 create_node + attach_node 执行 |
| 系统建议 | context 超预算时 | 脊髓引擎在组装认知包时检测到超预算，生成拆解建议注入 L_Alert |

## 5. 焦点切换时的 Context 保存与恢复

### 5.1 主动切换（用户/Agent 发起）

```
Step 1: 当前焦点的最新 context 已经通过 tool call 写回存储层了
Step 2: 旧焦点降级为 secondary（L1 摘要）
Step 3: 新焦点提升为 primary，重建 Context Bundle
```

不需要额外保存 — 因为写回已经通过 tool call 完成了。

### 5.2 被动打断（紧急告警抢占）

```
Step 1: 系统在 L_Alert 标记当前焦点被暂停
        格式: "暂停: [焦点title] — [最近一次 tool call 的结果摘要]"
Step 2: 暂停的焦点推入暂存区（stash, LIFO, max 2）
Step 3: 紧急焦点提升为 primary

打断条件（不是所有告警都打断）：
  - blocked 节点的 depended_by >= 3（爆炸半径大）
  - at_risk 且 deadline < 24h
  - 用户显式 shift_focus
```

### 5.3 恢复

```
紧急事项处理完毕后，系统在 L_Alert 提示：
  "待恢复: [暂停焦点title] — [暂停时的进展]"

Agent 决定是否 shift_focus 回去。系统不自动切回。

恢复时的 Context Bundle 额外包含：
  - 暂停原因
  - 暂停期间该节点的 narrative 新增条目（如果有）
```

### 5.4 暂存区衰减

```
暂存超过 24h:
  -> 将暂停快照写入节点 narrative（不丢失）
  -> 从 stash 移除
  -> 信息保留在节点叙事中，未来装载时自然可见
```

## 6. 审计问题修复

本文档同时澄清 PRD-functional-v4 审计中发现的关键问题：

### 6.1 派生层隔离 vs rollup 的矛盾

**澄清**：写路径"禁止读派生物"指的是禁止读 derived_cache 等缓存表。Rollup 冒泡是**事务内的确定性计算**，属于写路径的一部分，不是"读派生物"。

```
正确理解：
  写入 node status -> 触发 rollup 重算（事务内同步） -> 更新 parent status
  这整个过程是一个原子写事务，rollup 不读缓存，直接从 nodes 表计算

错误理解：
  写入 node status -> 读取 derived_rollup_cache -> 基于缓存更新 parent
  这才是违规的
```

### 6.2 done 是否为终态

**澄清**：done 是软终态。

- done -> archived：自动（7天 + 入度=0），不可逆
- done -> active：需要 reason，表示"重新打开"，是罕见操作
- archived：硬终态，只能通过 unarchive 恢复到 inbox

### 6.3 焦点丢失的降级行为

**澄清**：焦点丢失 = 进入无焦点模式。

- 认知包仅包含 L0 + L_Alert
- Agent 仍可通过 search_nodes 和 get_node 主动查询
- 等待用户 shift_focus 或 Heartbeat 产生新候选
