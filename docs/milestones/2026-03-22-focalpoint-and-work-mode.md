# FocalPoint and Work Mode — 设计思路演进记录

> 2026-03-22 | Jeff × Claude 深度讨论
> 从记忆引擎到 AI 认知操作系统的完整思考过程

---

## 起点：FPMS 已经做完了，然后呢？

FocalPoint（原名 FPMS）v0.2 已发布。记忆引擎完成：节点管理、状态机、Heartbeat、Context Bundle、GitHub/Notion 双向同步、MCP Server 18 tools、PyPI + ClawHub 发布。

584 tests 全绿。产品能用了。

但 Jeff 问了一个关键问题：**"市面上有类似的产品吗？我们和 Mem0、OpenViking 比有什么优势？"**

## 第一个转折：我们不是记忆工具

对比后发现，FocalPoint 和 Mem0/Zep/OpenViking 解决的是不同问题：

- Mem0 回答：**"上次聊了什么？"**
- FocalPoint 回答：**"你有 3 个任务 blocked，项目 X 已经 5 天没进展了"**

FocalPoint 的核心不是"记住对话"，是"管理正在做什么 + 主动提醒该做什么"。这是注意力管理，不是记忆存储。

## Work Mode 的灵感：人类怎么高效工作

Jeff 提出了一个观察：一个人高效完成任务的步骤是：

1. 定义任务的目标
2. 准备与任务相关的工具和材料
3. 拆解成一系列可完成的动作（考虑精度、体力、注意力、时间）
4. 开始工作
5. 每个细分动作验证是否和预期相同

AI 现在直接跳到第 4 步。没有准备阶段。这就是 Work Mode 要解决的问题 — 给 AI 一个"准备工作台"的阶段。

## 三省六部的启发

Jeff 提到中国古代三省六部制度可以借鉴：

- **中书省** → 决策（做不做？为什么做？）
- **门下省** → 审核（有没有风险？会不会踩坑？）
- **尚书省** → 执行（怎么做？怎么验收？）

每个角色只关注自己的职责，不越权。

### 关键洞察：每层 Context 不同

这个讨论的核心突破是：**不是所有 agent 都需要看到所有东西。** 中书省不需要看代码细节，尚书省不需要看市场分析。在有限的 token 预算里，装载最该关注的信息。

## 产品方向决策：做大还是做小？

讨论了两条路：

1. **独立**：三省六部是独立的工作流产品，FocalPoint 只是记忆插件
2. **结合**：FocalPoint 升级为 AI 认知操作系统

关键判断：**三省六部离开 FocalPoint 的记忆层还有意义吗？**

答案是没有。角色化上下文装载是三省六部的灵魂，这个能力只有 FocalPoint 的记忆层能提供。没有智能记忆层的工作流编排，和普通 prompt chain 没区别。

**决策：结合。FocalPoint = 记忆 + 注意力管理 + 工作流编排 = AI 认知操作系统。**

这就是 FounderOS 最初的愿景。绕了一圈，回到了起点，但手里有了实际跑通的代码。

## 角色视角的本质：不是过滤数据

最初的设计是按角色过滤数据 — strategy 角色不加载 technical 的 log，execution 角色不加载 decision 的 log。

Jeff 纠正了这个思路：**"记忆就是记忆，没有角度。角色带有的是看待同样数据的角度不同。"**

同一条记录"选了 Stripe"：
- 中书省看到 → 思考"这个决策对不对"
- 门下省看到 → 思考"Stripe 有没有历史教训"
- 尚书省看到 → 思考"知道了，按 Stripe API 实现"

**差异不在数据过滤，在角色的 system prompt。** 每个角色需要自己的认知框架文档。

## 知识文档层的发现

讨论中发现 FocalPoint 缺少一个关键层：项目的认知基底。

现在只记录执行层信息（状态、日志、依赖），但开发 FocalPoint 时反复要看的是 WhitePaper、PRD-philosophy、ARCHITECTURE — 这些是项目的知识基底，不是任务日志。

### 解决方案：Knowledge Layer

每个节点可以挂载知识文档：

```
data/knowledge/
└── {node_id}/
    ├── overview.md        → 是什么、为什么做
    ├── requirements.md    → 具体要做什么
    ├── architecture.md    → 怎么做的设计
    └── {自定义名}.md      → 可扩展
```

### 继承机制

子节点没有的知识文档，往上找父节点的。自己有的覆盖父节点的。

```
project → 有 overview + requirements + architecture
  └── milestone → 有自己的 architecture（覆盖）
       └── task → 什么都没有 → 继承 milestone 和 project 的
```

### narrative vs knowledge

讨论了"决定用 Stripe"应该存在哪。结论是**两个都存**：

- narrative：3月22日决定用 Stripe，原因是...（过程记录）
- knowledge：支付方案：Stripe（当前状态）

一个记过程，一个记结论。

## 七个漏洞的排查

系统性检查了两大模块的漏洞：

### 漏洞 1：节点之间的"软关联"

"支付系统"和"用户系统"经常一起讨论但没有依赖关系。要不要记录这种软关联？

**结论：不做。** 人脑的关联网络太复杂，计算机的优势是精确检索。不预建关联，需要时全文搜索。

### 漏洞 2：搜索能力弱

现在只能按字段过滤，不能搜"上次关于缓存的决策"。

**结论：加全文搜索（SQLite FTS5）。** 和漏洞 1 合并解决。

### 漏洞 3：知识文档类型太少

只有 overview/requirements/architecture 够吗？

**结论：基础三种 + 可扩展。** 允许自由命名，想加什么加什么。

### 漏洞 4 + 5：三省交接 protocol

中书省决策完怎么传给门下省？门下省通过后怎么传给尚书省？

**结论：并行审查。** 中书省产出需求文档后，门下省和尚书省同时审查。门下看历史教训和风险，尚书看工程可行性。两个都通过才执行。打回超过 3 次通知人类。

### 漏洞 6：工作台生命周期

什么时候创建？什么时候销毁？能开几个？

Jeff 的判断：工作台的本质是把任务拆分成一个 context 下能闭环的子任务。拉长没有意义。

**结论：无状态。** 工作台是一次函数调用，不是持久对象。调一次返回一次。

### 漏洞 7：narrative 和 knowledge 的边界

"决定用 Stripe"存哪？

**结论：两个都存。** narrative 记过程，knowledge 记结论。AI 做完决策后同时写两处。

## 三省角色的最终定义

讨论过程中经历了几次迭代：

1. 最初：三省串行（中书→门下→尚书）
2. 优化：并行审查（中书产出后，门下+尚书同时审）
3. 曾考虑砍掉门下省（个人创业者不需要合规）
4. 曾考虑砍掉尚书省（执行者就是 Claude）
5. 最终保留三省，但各自聚焦一件事

最终定义：

```
中书省（Maker）     → 需求决策：该不该做？做什么？
门下省（Reviewer）  → 经验教训：会不会踩坑？历史上出过什么事？
尚书省（Engineer）  → 工程评审：做得了吗？怎么做最优？+ 执行
```

**保留三省的核心原因：注意力管理。** 一个 agent 同时关注太多维度，精度就下降。这和 FocalPoint 的核心理念一致。

## 最终架构

```
FocalPoint = AI 认知操作系统

第一层：记忆引擎（✅ 已完成）
  节点管理、状态机、Heartbeat、Context Bundle
  GitHub/Notion 双向同步
  → 让 AI 记住事情

第二层：知识层 + 工作台（v0.3 要做的）
  知识文档（overview/requirements/architecture + 可扩展）
  继承机制（子节点继承父节点的知识）
  工作台（activate_workbench — 无状态，一次调用）
  三省 protocol（并行审查、≤3次打回、人类兜底）
  全文搜索
  → 让 AI 关注对的事情

第三层：协作层（未来）
  多 Agent 任务分配和协调
  跨 Agent 共享记忆
  → 让多个 AI 像团队一样协作
```
